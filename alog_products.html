🤔 そもそも「ログ管理」って何？

📋ログとは？

  • コンピュータやシステムの動作記録
  • 「いつ」「誰が」「何をした」が記録される
  • 例: ファイルを開いた、ログインした、等
  • セキュリティや監査で非常に重要

❓なぜログ管理が必要？

  • セキュリティ: 不正アクセスの検知
  • 内部統制: 情報漏洩の追跡
  • トラブル対応: 障害原因の調査
  • 監査対応: 法令・規制への対応

🏢お客様がログ管理を必要とする場面（実例）

セキュリティ事故

「退職者が顧客情報を持ち出したかも...」→ ファイルアクセスログで調査

コンプライアンス

「監査でログの提出を求められた」→ レポート機能で対応

障害対応

「システムが突然止まった！」→ ログから原因を特定

📌 一言でまとめると: ログ管理製品 = 企業の「防犯カメラ」のようなもの。何か起きた時に「誰が何をしたか」追跡できる

ALog（エーログ） 🌟 主力製品

📖これだけは覚えよう

  • 当社のメイン製品・看板商品
  • 様々なシステムのログを一元管理
  • ログの収集→保存→検索→分析→レポートが可能
  • 中〜大規模企業がメインターゲット

🎯どんなログを集められる？

  • Windows: ファイルサーバー、AD、イベントログ
  • NAS: 各社ファイルサーバーのアクセスログ
  • データベース: SQL Server、Oracle等
  • クラウド: Microsoft 365、Box等

💼よくある導入理由

  • 「ファイルサーバーの誰が何を見たか知りたい」
  • 「内部不正を監視・抑止したい」
  • 「監査対応でログ提出が必要」
  • 「J-SOXやPマーク対応したい」

⚙️提供形態

  • オンプレミス版: お客様環境に構築
  • クラウド版: SaaS形式で提供
  • ※両方を組み合わせるハイブリッド構成も可能

🔑 新人が覚えるべきポイント

ALogは「ファイルサーバーのアクセスログ管理」で特に強い。「誰がいつどのファイルを開いた/編集した/削除した」がわかる！

ALog SMASH（スマッシュ） 💰 廉価版

📖これだけは覚えよう

  • ALogの廉価版・簡易版
  • 中小企業や部門単位での導入向け
  • 低価格で導入しやすい
  • 機能を絞ってシンプルにした製品

🔄ALogとの違い

  • 対応ログの種類が限定的
  • 高度な分析機能は省略
  • 大規模環境には非対応
  • その分価格が安い！

👥こんなお客様向け

  • 「まずはお試しでログ管理を始めたい」
  • 「予算が限られている」
  • 「IT担当者が少ない/いない」
  • 「最低限の監視で十分」

📈アップグレードパス

  • SMASHで始めて、後からALogに移行可能
  • 蓄積したログデータも引き継げる
  • 操作感が同じなので移行がスムーズ
💡 営業的な位置づけ: 「ALogは高い...」と言われたら SMASH を提案。スモールスタートしてもらい、将来的にALogへアップセルするパターンも多い

Resource Athlete（RA） 📊 リソース監視

📖これだけは覚えよう

  • サーバーのリソース（CPU/メモリ/ディスク）を監視
  • ログ管理ではなく「監視」製品
  • ALogと組み合わせて使うことが多い
  • 略称「RA」で呼ばれることが多い

👀何を監視できる？

  • CPU使用率: 処理が重くないか
  • メモリ使用率: メモリ不足になっていないか
  • ディスク容量: 空き容量は足りているか
  • ネットワーク: 通信量は正常か

🤔ALogとの違いは？

  • ALog: 「誰が何をした」を記録（操作ログ）
  • RA: 「今どんな状態か」を監視（状態監視）
  • 目的が違うので補完関係にある
  • 両方セットで導入するお客様も多い

💼よくある導入理由

  • 「サーバーが重くなる前に気づきたい」
  • 「ディスク容量がいっぱいになる前に警告がほしい」
  • 「障害の予兆を検知したい」

🔑 新人が覚えるべきポイント

RA = サーバーの健康診断ツール。ALogが「行動記録」なら、RAは「体温・血圧測定」のイメージ！

ALog Syslog Receiver（ASR） 📡 ログ収集

📖これだけは覚えよう

  • Syslogというプロトコルでログを受信するツール
  • ネットワーク機器やLinuxのログ収集に使う
  • ALogの補助ツール的な位置づけ
  • 略称「ASR」で呼ばれる

🤔Syslogとは？

  • ネットワーク機器がログを送る標準規格
  • ファイアウォール、ルーター、スイッチなどが対応
  • Linuxサーバーも標準でSyslog出力可能
  • ※WindowsはSyslogを使わない（別の形式）

🔧なぜ必要？

  • ALog単体ではSyslogを直接受信できない
  • ASRがSyslogを受け取り、ALogに渡す役割
  • いわば「通訳」のような存在

🌐対象となる機器

  • ファイアウォール: Fortinet、Palo Alto等
  • ネットワーク機器: Cisco、Juniper等
  • Linuxサーバー: CentOS、Ubuntu等
  • 各種アプライアンス
📌 一言でまとめると: ASR = ALogの「Syslog専用の入り口」。ネットワーク機器のログをALogで管理したい時に必要になる

製品 一言で言うと ALogとの関係 覚えておくキーワード
ALog 統合ログ管理の中核 —（これが本体） ファイルサーバー操作ログ
SMASH ALogの廉価版 機能を絞った別製品 低価格中小企業
RA リソース監視ツール ALogと組み合わせ CPU/メモリ状態監視
ASR Syslog収集ツール ALogの補助ツール SyslogNW機器

🔗 よくある導入パターン

1️⃣ALog単体

最もベーシックな構成

  • ファイルサーバーのログ管理がメイン
  • Windows環境中心のお客様
  • まずはログ管理から始めたい

2️⃣ALog + RA

ログ管理＋監視の人気構成

  • ログ管理とリソース監視を両立
  • 障害発生時の調査が楽になる
  • 運用管理を一元化したいお客様

3️⃣ALog + ASR

ネットワーク機器も対象の構成

  • ファイアウォール等のログも収集
  • Linuxサーバーがある環境
  • セキュリティ重視のお客様

🏆フル構成: ALog + RA + ASR

ログ管理（ALog）+ リソース監視（RA）+ Syslog収集（ASR）で、
IT環境全体を包括的に管理できる最も充実した構成。大企業やセキュリティ重視のお客様向け。

💡 SMASHの場合: SMASHは単体での利用が基本。RA/ASRとの組み合わせは、ALogにアップグレードしてから検討することが多い
